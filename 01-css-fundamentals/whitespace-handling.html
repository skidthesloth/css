Understanding Whitespace Sensitivity in CSS

CSS is generally quite forgiving when it comes to whitespace. This means you have a lot of flexibility in how you format your stylesheets, which can greatly improve readability and maintainability. Let's explore how CSS handles whitespace, both between and within rules.

Whitespace within CSS Rules

CSS is largely insensitive to whitespace within rules. Any sequence of whitespace characters (spaces, tabs, newlines) is collapsed to a single space for parsing purposes. This allows for various formatting styles without affecting how the browser interprets the CSS.

Example: The "Rainbow" Rule

Consider a hypothetical CSS property rainbow with a list of values. You can format this rule in multiple ways, all of which are valid: 

/* All on one line */
rainbow: infrared red orange yellow green blue indigo violet ultraviolet;

/* Values on a new line */
rainbow:
infrared red orange yellow green blue indigo violet ultraviolet;

/* Each value on its own line */
rainbow:
infrared
red
orange
yellow
green
blue
indigo
violet
ultraviolet
;

As you can see, the only restriction is that the separating characters must be whitespace: an empty space, a tab, or a newline, alone or in combination, as many as you like.

Whitespace between CSS Rules

Similarly, you can format a series of CSS rules with whitespace in any fashion you like. The browser will parse them correctly regardless of the spacing. Here are five examples demonstrating this flexibility: 

/* Minimal whitespace */
html{color:black;}

/* Standard formatting */
body {background: white;}

/* More spaced out */
p {
    color: gray;
}

/* Even more whitespace, with indentation */
h2 {
    color : silver ;
}

/* Highly verbose formatting, each part on a new line */
ol
{
    color
    :
    silver
    ;
}

The first rule demonstrates how whitespace can be largely omitted. This is a common practice in minified CSS, where all possible extraneous whitespace is removed, usually by automated server-side scripts, to reduce file size and improve loading performance.

The subsequent rules progressively use more extravagant amounts of whitespace, illustrating that the parser is robust enough to handle it. In the last rule, almost every part of the declaration is placed on its own line.

All of these approaches are valid. The best practice is to choose a formatting style that makes the most sense for you and your team—one that is easiest to read and maintain—and stick with it consistently throughout your stylesheets.
