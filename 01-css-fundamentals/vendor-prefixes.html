Understanding Vendor Prefixes

Vendor prefixes are special labels, often with hyphens, that browser vendors used to add to experimental or nonstandard CSS properties and JavaScript APIs. For example, you might see -webkit-transition or -moz-transform.

What Are Vendor Prefixes?

They typically follow the format of a hyphen, a label (like webkit), and another hyphen (e.g., -webkit-). They were a way for browser makers to introduce new features that were still under development or were unique to their browser.

Why Were They Used?

  • Experimentation: They allowed browser vendors to test new CSS properties and JavaScript APIs in real-world scenarios.
  • Preventing Lock-in: The idea was to let developers experiment without the browser being locked into a specific, potentially changing, behavior before a feature became a standard. This helped prevent compatibility issues that could arise if a feature changed after widespread adoption.

Common CSS Vendor Prefixes (Historically)

You might find these in older codebases:

Prefix Vendor
-webkit- WebKit-based browsers (e.g., Chrome, Safari, newer Opera and Edge, most iOS browsers)
-moz- Gecko-based browsers (e.g., Mozilla Firefox)
-o- Older, pre-WebKit versions of Opera
-ms- Internet Explorer and Microsoft Edge (before it switched to Chromium)
-epub- International Digital Publishing Forum ePub format

Example CSS Usage (Older Code - All instances of vendor prefixes are shown below):

-webkit-transition: all 4s ease; /* For WebKit browsers */
-moz-transition: all 4s ease;    /* For Firefox */
-ms-transition: all 4s ease;     /* For Internet Explorer/Edge */
-o-transition: all 4s ease;      /* For Opera */
transition: all 4s ease;         /* The standard way */

Modern Equivalent:

Today, you only need the standard property:

transition: all 4s ease;

API Prefixes

Vendor prefixes were also applied to JavaScript APIs:

  • Interface Prefixes: If an entire API interface was experimental, its name was prefixed with an uppercase prefix (e.g., WebKit, Moz, O, MS).
  • Property and Method Prefixes: If a specific property or method within a standard interface was experimental, it used a lowercase prefix (e.g., webkitRequestAnimationFrame).

Example JavaScript Usage (Older Code - All instances of vendor prefixes are shown below):

window.requestAnimationFrame =
  window.requestAnimationFrame ||
  window.mozRequestAnimationFrame ||
  window.webkitRequestAnimationFrame ||
  window.oRequestAnimationFrame ||
  window.msRequestAnimationFrame;

Modern Equivalent:

Today, the standard method is widely supported:

requestAnimationFrame(callback);

Why They Are No Longer Used (Mostly)

While vendor prefixes aimed to help, they created new problems:

  • Widespread Adoption of Experimental Features: Developers often used prefixed features on production websites, even though they were experimental. This meant that when standards changed, these websites could break.
  • Burden on Smaller Browsers: Smaller browser vendors were forced to implement other browsers' prefixes to ensure compatibility with popular websites, which was a significant overhead.

Current Approach:

As of early twenty twenty-three, vendor-prefixed CSS features are rare. Instead, experimental features are now typically "put behind a flag" in browser settings. This allows developers to manually enable and test upcoming features without affecting the general web, leading to faster standardization and fewer compatibility issues.

You are unlikely to write new code with vendor prefixes today, but you might encounter them in older or legacy codebases. In most cases, you can safely remove the prefixed versions and keep only the standard property or method.
